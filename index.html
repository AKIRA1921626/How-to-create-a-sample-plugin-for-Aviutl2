2. プラグイン開発における言語と環境の前提

プラグインは、C++などのプログラミング言語で作成したコードを「コンパイル（パソコンが実行できるファイル形式に変換すること）」することで利用可能になります 。C#、C言語、Rust、Delphiなどでも作成可能なようですが、公式サンプルのプラグインがC++で記述されているため、今回はC++を採用します （Lua単体でのプラグイン出力はできません ）。C++によるコンパイルを最も簡単に行う手段として、Visual Studioを利用するのが合理的です 。

3. 開発環境のインストール手順

  1. VisualStudioSetup.exe を起動します 。(これはVisual Studio 2026をダウンロードした時に入手可能なインストールファイルです。)

  2. 「続行」をクリックします 。

  3. インストールの詳細画面が表示されたら、「デスクトップとモバイル」項目内にある「C++によるデスクトップ開発」を選択します。このとき、インストールの詳細は変更を行わずにデフォルトのままとします 。

  4. 「インストール」をクリックし、完了まで待ちます（20分程度かかります） 。
  5. インストール完了後、Visual Studioを起動します 。

  6. Microsoftアカウント、またはGitHubアカウントでのログインが要求されるため、ログインします（後から変更やスキップも可能です） 。

  7. 開発設定（変更なし）と配色テーマ（任意）を選択して開始します 。

4. プロジェクトの新規作成と初期設定

  1. 起動画面から「新しいプロジェクトの作成」を選択します 。

  2. テンプレート一覧から「ダイナミック リンク ライブラリ (DLL)」を選択し、「次へ」をクリックします 。

  3. プロジェクト名を任意のものに変更し（今回は Sample とします）、それ以外は変更せず「作成」をクリックします。画面に表示される dllmain.cpp が、後の Sample.dll の中身となります 。

  4. プロジェクトが開かれたら、画面上部のツールバーにある「Debug」の隣を確認し、x86 であればx64に変更します（最初から x64 の場合はそのままにします）。

    5. 【解説】なぜx64にする必要があるのか？

    ● プラットフォーム（ビルド対象）とは？
    これは、作成するプログラムを「32bit環境（x86）」と「64bit環境（x64）」のどちらに向けてビルド（出力）するかを指定する設定です。Visual Studioで新しいプロジェクトを作成すると、初期設定が x86 になっている場合があります。

    ● 変更しないとどうなるか（エラーや不具合の原因）
    x86 のままビルドを行うと、32bit用のプログラムが出力されます。
    しかし、AviUtl2はシステム全体が64bit専用に設計されているため、32bit用に作られたプラグイン（.auc や .aui など）を認識することができません。そのため、ビルド自体は成功したとしても、AviUtl2のプラグインフォルダに配置した際に「プラグインが一切読み込まれない（一覧に表示されない）」という問題が発生します。

    ● 設定を変更する理由
    AviUtl2で動作するプラグインを開発するためには、出力されるバイナリが64bit形式である必要があります。AviUtl2用のプラグイン開発においては、開発の最初期段階でプラットフォームを『x64』へ明示的に変更することが必須条件となります。

    なお、従来の「AviUtl（旧バージョン）」は32bitソフトであったため x86 で開発されていましたが、現在の「AviUtl2」は64bitソフトであり互換性がないため混同しないよう注意が必要です。

  5. 次に、不要なエラーを防ぐ設定を行います。画面右下からソリューションエクスプローラーの項目を開き、プロジェクトを右クリックして「プロパティ」を開きます。

  6. この時、開いたプロパティの「構成」が「すべての構成」になっているかを確認してから行ってください。

    7. 【解説】なぜ「すべての構成」にする必要があるのか？

    ● 「構成」の種類とは？
    Visual Studioには、開発中のデバッグ（動作確認）に適した Debug 設定と、最終的な配布用ファイルを出力する Release 設定の2種類が最初から用意されています。

    ● 「すべての構成」にしないとどうなるか
    ここが「Debug」のまま設定を変更した場合、変更内容はDebug用にしか保存されません。
    本手順の後半（手順6の3）では、プラグインを出力するために設定を Release に切り替えてビルドを行います。そのため、ここで「すべての構成」を選んでおかないと、Releaseに切り替えた時点でプリコンパイル済みヘッダーの機能が「有効」に戻ってしまい、最終的なビルドの段階で致命的なエラー（C1010など）が発生してしまいます。

    ● 設定を共通化する理由
    開発環境（Debug）でも本番出力（Release）でも、今回のコードをそのまま動かすためには「プリコンパイル済みヘッダーを使用しない」という条件は共通です。手戻りや設定漏れによるエラーを防ぐため、最初から『すべての構成』に対して一括で設定を適用させる必要があります。

  7. そして、「構成プロパティ」→「C/C++」→「プリコンパイル済みヘッダー」 を選択します。一番上の「プリコンパイル済みヘッダー」の項目を「プリコンパイル済みヘッダーを使用しない」に変更し、OKボタンを押します。後にサンプルコードをそのまま貼り付けて動かす際に、システム特有のエラーが出るのを防ぐために必要な設定です。

    8. 【解説】なぜこの設定変更が必要なのか？

    ● プリコンパイル済みヘッダーとは？
    これは、書き換えることのない巨大なシステムファイルをあらかじめ裏で読み込んでおくことで、2回目以降のビルド（ファイル変換）を高速化するVisual Studioの機能です。新しいプロジェクトを作ると、デフォルトでこの機能が「有効（使用する）」になっています。

    ● 行わないとどうなるか（エラーの原因）
    この機能が有効なままだと、Visual Studioは「すべてのプログラムの最優先事項として、システム専用のヘッダーファイル（pch.h や stdafx.h）を読み込みなさい」というルールでビルドを行います。
    しかし、手順6で貼り付ける公式のサンプルコード（AviReader.cpp）には、当然そんな特殊なファイルを読む指示は書かれていません。そのため、そのままビルドをしようとすると、『ルール通りに先頭でシステムヘッダーが読み込まれていません』という致命的なエラー（エラーコード: C1010 など）が発生し、ビルドが強制終了してしまいます。

    ● 設定を変更する理由
    今回の目的は「既存のサンプルプログラムをまずはそのまま動かすこと」です。処理するファイルが1つだけのような小さなプログラムでは、この高速化機能の恩恵は1秒未満しかありません。メリットに対して「エラーを引き起こす罠」になるデメリットの方が大きいため、安全のために機能を『使用しない』へと明示的にOFFにする必要があります。

    Aviutl2のプラグインのようなソースコードが少ないプログラムの場合、実際に独自のコードを作成する場合もOFFにしておく構成が一般的です。

    5. SDKヘッダーファイルの組み込み

    1. ダウンロードした aviutl2_sdk.zip を解凍しておきます 。

    2. VS2026に戻り、画面右側の「ソリューション エクスプローラー」から作成したプロジェクト（Sample）を右クリックし、「エクスプローラーで開く」を選択します 。

    3. 開いたプロジェクトのディレクトリ内に、解凍したSDK内にある拡張子 .h のファイルをコピーします 。今回は入力プラグインを対象とするため、input2.h をここにコピーします 。

      4. 【解説】なぜこのファイル（input2.h）を選択するのか？

      ● SDKに含まれるヘッダーファイル（.h）の種類
      AviUtl2のSDKには、開発するプラグインの役割（機能）に応じて、それぞれ異なるヘッダーファイルが用意されています。主なものは以下の通りです。

      • input2.h ：映像や音声のファイルを読み込むための「入力プラグイン」用
      • output2.h ：編集した動画をファイルとして書き出すための「出力プラグイン」用
      • filter2.h ：色調補正やエフェクトなどの効果を与えるための「フィルタプラグイン」用

      上記以外にも複数の.hファイルが存在します。

      ● 間違ったファイルをコピーするとどうなるか
      例えば、入力プラグインを作りたいのに filter2.h をコピーした場合、プログラム側から「入力プラグインを構成するための専用の命令や仕組み」を呼び出すことができなくなります。その結果、コード内にエラーが発生してビルドに失敗するか、ビルドが通ってもAviUtl2側で正しく機能が認識されません。

      ● 今回 input2.h を選ぶ理由
      今回の目的は「動画や音声を読み込むプラグイン（入力プラグイン）のサンプル」を作成することです。そのため、その開発の土台となる input2.h を選択してプロジェクトに配置する必要があります。将来的に出力プラグインやフィルタプラグインを自作する場合は、この段階でそれぞれの目的に応じたヘッダーファイルを選択することになります。

    4. VS2026のソリューションエクスプローラー上で、プロジェクトを右クリック ＞「追加」 ＞「既存の項目」を選択し、先ほどコピーした input2.h をプロジェクトに追加します 。

      【解説】なぜフォルダにコピーした後に、VS上で再度「追加」するのか？

      ● 「フォルダの中身」と「プロジェクトの管理対象」は独立している
      Windowsのエクスプローラー上でファイルをフォルダの中に移動させただけでは、Visual Studioはそのファイルが存在することに気づくことができません。Visual Studioは、自身が管理する「プロジェクト構成ファイル」に登録されているファイルだけをビルド（コンパイル）の対象とする仕様になっているためです。

      ● なぜ追加しなくても動くことがあるのか？
      C++のプログラム（.cpp）の中で #include "input2.h" と書いた際、Visual Studioは「プロジェクトに登録されているか」に関わらず、まずそのソースファイルと同じフォルダ内を自動的に探しに行く仕様になっています。そのため、フォルダにファイルが置いてありさえすれば、VS上に登録していなくてもビルド自体は成功してしまいます。

      ● それでも「追加」を推奨する理由
      ビルドが通るのになぜ追加するのか。それは、VSに登録されていないファイルは「検索や置換の対象外」になったり、「コードの自動補完（IntelliSense）が効かなくなる」など、開発時のサポート機能が制限されるためです。
      また、将来的にファイルを別のフォルダへ整理した際、VSに登録していないファイルは追跡できず、その段階で初めて致命的なビルドエラーを引き起こします。安全かつ快適に開発を続けるための「保険」として、明示的にプロジェクトへ登録（追加）しておくことが推奨されます。

    6. サンプルコードの実装とビルド

    1. SDK内のサンプルソースである AviReader.cpp を開き、その全内容をコピーします 。

      2. 【解説】なぜこのファイル（AviReader.cpp）をコピーするのか？

      ● サンプルソース（.cpp）とは？
      SDK（開発キット）の中には、プラグインの土台となるプログラムの記述例（サンプルソース）がいくつか同梱されています。これらは、特定の機能を実装するために必要な最低限のコードがあらかじめ記述された「設計図の雛形」です。

      ● 別のサンプルを使用したり、一から書こうとするとどうなるか
      SDKには他にも出力用やフィルタ用のサンプルが含まれています。目的と異なるサンプルをコピーしてしまうと、前述の手順で用意した input2.h と内部の命令が噛み合わず、大量のビルドエラーが発生します。
      また、これらを手動で一から記述する場合、AviUtl2の内部システムと通信するための複雑な初期化処理や構造体の定義をすべて正確に書き起こす必要があり、開発の難易度が極端に上がってしまいます。

      ● 今回 AviReader.cpp を選ぶ理由
      このファイルは、動画ファイルを読み込むための「入力プラグイン」の最も標準的なサンプルコードです。手順7で配置した input2.h と完全にセット（一対の構成）になっており、そのままビルドするだけで正しく動作するプラグインを出力できるよう設計されています。
      そのため、まずはこの正常に動く雛形をプロジェクトにそのまま取り込み、ビルドができる環境が整っているかを確認するための基準点として選択しています。

    2. VS2026上の dllmain.cpp の中身を、コピーした内容で完全に上書きします 。このコードは、先ほど追加した input2.h とシステム用の windows.h を読み込む、入力プラグインのサンプルロジックです 。

    3. ビルドを行うため、ツールバーの「Debug」「x64」となっている箇所のドロップダウンを、「Release」「x64」に変更します 。

      4. 【解説】なぜ「Release」に変更するのか？

      ● 開発用から「配布用」への切り替え
      手順4で解説した通り、Debug は開発中のテストに適したモードであり、不要なデータが含まれるためファイルサイズが大きくなります。
      今回は最終成果物となるプラグイン（Sample.aui2）を出力することが目的であるため、プログラムが最適化され、無駄のない軽量なファイルが生成される Release モードへと明示的に切り替える必要があります。

    4. メニューから「ビルド」 ＞ 「ソリューションのビルド」を実行します 。

    5. ビルド完了後、プロジェクト名を右クリックして「エクスプローラーで開く」を選択し、表示されたディレクトリの1つ上の階層にある x64 フォルダを開きます 。

    6. その中の Release フォルダ内に、プロジェクト名と同名の Sample.dll が生成されていることを確認します 。

    7. 配置と動作確認

    1. 生成された Sample.dll の拡張子を手動で変更し、入力プラグインの形式である Sample.aui2 に書き換えます 。

      2. 【解説】なぜ拡張子を「.aui2」に書き換えるのか？

      ● 拡張子（.dll と .aui2）とは？
      Visual Studioがプログラムをビルドした際、初期設定では汎用的な動的ライブラリを示す .dll という拡張子でファイルが出力されます。一方、.aui2 はAviUtl2が「これは入力プラグインである」と識別するための専用の拡張子です（AviUtl Input の略）。

      ● 変更を行わないとどうなるか
      出力された Sample.dll をそのままAviUtl2のプラグインフォルダに配置した場合、AviUtl2はそれをプラグインとして認識しません。ビルド自体は100%成功しており、プログラムの内部構造が正しくても、拡張子が .dll のままであるという理由だけで、システムに無視されて読み込まれない状態になります。

      ● 手動で書き換える理由
      Visual Studioに最初から .aui2 として出力させる設定も可能ですが、プロジェクトの設定を書き換える手間が発生します。そのため、最もシンプルかつ確実な方法として、出力された .dll ファイルの名前（拡張子）をエクスプローラー上で直接書き換える手順をとっています。

      なお、作成するプラグインの種類によって、最終的に変更すべき拡張子は以下のように異なります。

      • 入力プラグイン：.aui2（AviUtl Input 2）
      • 出力プラグイン：.auo2（AviUtl Output 2）
      • フィルタプラグイン：.auf2（AviUtl Filter 2）
    2. 変更したファイルを、AviUtl2のプラグイン認識ディレクトリ（例：C:\ProgramData\aviutl2\Plugin）にコピーします 。

    3. AviUtl2を起動し、メニューの「その他」 ＞ 「プラグイン情報」を確認します 。

    4. 一覧に「AVI File Reader(sample)」が表示されていれば、サンプルプラグインのビルドと組み込みは成功です 。

      5. このプラグインは元からあるものではなく、今までの操作で新たに導入したプラグインです。 先ほどプラグインを認識するディレクトリに配置した「Sample.aui2」を削除し、Aviutl2を再起動してみれば、このプラグインは消失することが確認できます。

    8. プラグイン情報の変更と再ビルド

    1. プラグイン名や識別情報を固有のものに変更するため、VS2026で dllmain.cpp を開きます 。

    2. 「入力プラグイン構造体定義」が記述されているセクションを探し、文字列の値を任意の内容に書き換えます 。

      3. 注意点：
      文字列の先頭にある L および ""（ダブルクォーテーション）は削除しないでください 。L は型を「ワイド文字列」というAviUtl2のAPI仕様に適応させるための識別子であり、削除すると型不一致によるコンパイルエラーが発生します。
    3. 下記の作業を行う前に手順6.3〜7.4で作成したSample.aui2 を事前に削除していないと、名前が被って下記の項目は行えないので削除しておきます。

      4. もしも何らかの理由で、このディレクトリにSample.dllを保持していた場合、Sample.dllの方は、ソリューションのビルドを行えばその時に上書きされるため、削除する必要はありません。（してもかまいません。）

    4. その後、再度手順6.3〜7.4と同様の手順で「Release / x64」ビルドを行い、拡張子を .aui2 に変更します 。

    5. 変更したファイルを、再度AviUtl2のプラグイン認識ディレクトリ（例：C:\ProgramData\aviutl2\Plugin）に上書き、または、コピーします 。

    6. AviUtl2を再起動し、プラグイン情報に変更した名称が反映されていることを確認します 。

    7. これでAviUtl2のサンプルプラグインの作成と動作確認は完了となります。
